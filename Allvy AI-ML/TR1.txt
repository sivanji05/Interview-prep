🔹 What is RAG?

RAG = Retrieval-Augmented Generation
It’s an AI architecture that combines retrieval (finding relevant information) with generation (LLM producing text) to give accurate, grounded answers.

Instead of relying only on what a Large Language Model (LLM) knows (which may be outdated or hallucinated), RAG lets the model search external knowledge sources (databases, documents, Elasticsearch, etc.) and then generate an answer based on that retrieved context.

🔹 How RAG Works (Step by Step)

Document Ingestion

Documents are parsed, chunked, and stored with embeddings in a vector database / search engine (Elasticsearch in your project).

User Query

The user asks a question: e.g., “What are the payment terms in this contract?”

Retrieval

The query is converted into an embedding.

Similar chunks from stored documents are retrieved using semantic search.

Augmentation

Retrieved chunks are added to the prompt as context.

Generation

LLM (via LangChain/Autogen in your project) uses both the user’s question + retrieved context to generate an accurate, grounded answer.

🔹 Why RAG is Important

✅ Prevents hallucinations → answers come from your documents, not just the model’s memory.

✅ Domain-specific knowledge → works even if the model wasn’t trained on your private data.

✅ Scalable → just keep adding documents; retrieval adapts.

✅ Cost-efficient → avoids retraining LLMs.



3.What Happens If You Increase the Overlap?

Better Context Preservation ✅

Important when concepts span across chunk boundaries.

Higher overlap helps the retriever capture continuity.

Example: a legal clause that starts in one chunk and ends in the next.

Improved Recall ✅

Search results are more likely to return relevant chunks, since overlapping increases redundancy.

Reduces the chance of “cut-off” answers.

Higher Storage & Embedding Cost ❌

More overlap → more tokens repeated → more embeddings stored in Elasticsearch.

Index size grows.

Slower Retrieval ❌

Larger index = slightly slower searches.

More chunks retrieved to process.

Risk of Repetition in LLM Answer ❌

Since overlapping text may appear in multiple retrieved chunks, LLM could repeat content in responses.

🔹Rule of Thumb

Small overlap (10–20%) → balances context + efficiency.

Too high overlap (>50%) → wastes space, adds redundancy.

Too low (0%) → risks cutting off meaning at boundaries.

Handling Images in Documents

Image Detection

unstructured.io identifies embedded images in PDFs/PPTs.

For metadata, we store the image reference (filename + page location) in PostgreSQL/S3.

Text in Images (OCR)

If images contain text (e.g., scanned contracts, invoices), we run OCR (Tesseract or AWS Textract) to extract textual content before chunking.

Non-text Images (Charts, Diagrams, Logos)

Currently, these are either:

Stored as attachments (S3 reference) with metadata.

Optionally processed using Vision-Language Models (e.g., BLIP, Donut, GPT-4V) if advanced analysis is required.

User Query Flow

If a question targets images (e.g., “What does the chart on page 5 show?”), system:
